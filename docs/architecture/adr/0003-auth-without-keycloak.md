# ADR 0003: Auth without Keycloak

## Статус

Принято.

## Контекст

Во второй версии проекта рассматривался Keycloak. Он дает много возможностей, но
для текущего проекта добавляет отдельную инфраструктуру и усложняет разработку.

## Решение

На старте не используем Keycloak. Аутентификацию и авторизацию реализуем внутри
FastAPI-приложения.

Планируемая модель:

- пользователи хранятся в PostgreSQL;
- пароли хранятся как безопасные хеши;
- login использует access token и refresh token;
- регистрация требует подтверждения email;
- админские действия защищены ролями и областями доступа.

## Последствия

- Меньше инфраструктуры на старте.
- Полный контроль над UX регистрации и администрирования.
- При появлении требований SSO или внешних identity providers решение можно
  пересмотреть отдельным ADR.
